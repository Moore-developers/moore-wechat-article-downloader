# 微信收藏架构设计

## 结论

将面向用户的“代理增强模式”更名为**微信收藏**。代理、端口、证书和 WebView 重置全部降为内部实现细节。

产品只保留两个主场景：

1. **订阅同步**：用 Exporter 搜索公众号、维护订阅、同步历史文章和定期增量更新。
2. **微信收藏**：用户在微信阅读时收藏当前文章；在短时凭证有效期内，可为同一公众号的已同步文章批量补充互动指标和精选评论。

“直接下载 URL”不再是与前两者并列的用户模式，而是资料库写入正文时可选的获取策略和兜底入口。旧代理历史能力并入微信收藏，成为 Exporter 和短时凭证主动同步都不可用时的兼容兜底。

架构按职责分为四层：Exporter 是文章发现和任务规划的**控制面**；公开 URL 下载是正文获取的**内容面**；常驻微信收藏是凭证、页面快照和互动数据的**数据面**；本地资料库是唯一交付层。正文下载不等待短时凭证，互动数据在凭证可用后并行补充。

本项目只管理本地文章库和结构化数据。这里的“Wiki”是另一个系统；本项目不实现外部 Wiki 的上传、同步或权限管理。

## 术语

| 用户看到的名称 | 内部名称 | 作用 |
| --- | --- | --- |
| 订阅同步 | Exporter runtime | 公众号发现、订阅、历史列表和增量同步 |
| 微信收藏 | WeChat collection session | 微信桌面端阅读收藏、短时凭证、互动数据、历史采集备用 |
| 链接导入 | URL delivery pipeline | 已知 URL 的正文和图片归档 |
| 本地资料库 | article library | Markdown、图片、索引、SQLite 和结构化快照 |

不得在 README、Skill 或用户提示中使用“代理增强模式”“代理快照模式”作为主功能名。旧 CLI 名称可保留为兼容别名；诊断输出可以继续使用技术名。

## 产品边界

### 订阅同步

**用户目标**：长期关注一批公众号，查看新文章并决定哪些进入本地资料库。

- 使用 Exporter 的公众号后台授权；这是后台授权，不等同于微信桌面端登录。
- 授权用于搜索、订阅、同步文章元数据和选择范围。
- 定时同步默认只更新列表和文章状态，不默认拉取互动数据，也不自动下载全部正文。
- 用户可明确选择最新 N 篇、日期范围、标题关键词或指定文章进入资料库。

### 微信收藏

**用户目标**：正常阅读时保存当前文章；必要时为正在关注的公众号补齐短时互动数据。

- 目标状态是在常驻守护完成真实 macOS 故障验证后，本地服务和系统路由默认常驻；在此之前继续使用显式启动的手动收藏会话。
- 注入按钮“收藏到本地”保存当前文章正文、页面已加载评论和可观察指标。
- 打开目标公众号任意一篇文章，收藏会话捕获并仅短时保存该公众号的会话凭证。
- 凭证有效时，用户可明确请求“补充这个公众号最近 50 篇的互动数据和精选评论”。
- 收藏按钮提供当前文章的高保真页面证据；批量互动同步提供同公众号多篇文章的覆盖面。两者互补，不把“已加载评论”误称为完整评论。
- 旧 `profile_ext/getmsg` 被动捕获不再作为普通用户路径，仅作为兼容和诊断兜底。

### 链接导入

**用户目标**：把已有文章链接或 URL 列表放进本地资料库。

- 公开 URL 默认不需要任何授权或微信会话。
- 这是一个低摩擦备用入口，不承担公众号发现、订阅、互动数据或评论抓取。
- 当微信收藏保存的文章尚未存在于资料库时，收藏结果本身即可创建 Markdown；无需要求用户再执行“下载”。

## 任务编排

### 下载某公众号最新 20 篇

1. Exporter 同步公众号最新文章，确定目标 20 篇并在聊天中展示标题和日期。
2. 用户已经明确说“下载 20 篇”时，标题列表是进度和范围回显，不再增加一次无意义确认；只有目标范围含糊时才让用户选择。
3. 公开正文下载立即开始，不等待微信短时凭证。
4. 系统同时检查该公众号 credential 状态：`valid`、`expiring`、`missing` 或 `expired`。
5. 凭证有效时，批量互动任务并行抓取可用指标、精选评论和有限回复。
6. 凭证无效时，从目标文章中选择一篇代表 URL 复制到剪贴板，提示用户粘贴到微信文件传输助手并打开。微信桌面端没有可靠的通用地址栏，不应提示“直接粘贴到浏览器地址栏”。
7. 常驻微信收藏捕获新凭证后自动恢复等待中的互动任务，无需重新执行正文下载。
8. 最终报告分别给出正文成功数、互动成功数、缺失字段、精选评论范围和失败项。

### 拆解最新 20 篇并生成选题

本 Skill 只负责准备本地数据集，不负责内容拆解或生成选题。

1. 复用上述同步、正文下载和互动补充流程。
2. 生成稳定的数据集清单，包含文章路径、标题、日期、互动指标、精选评论路径和数据完整性。
3. 下游 Wiki/分析机制读取本地清单和 Markdown，执行文章拆解、模式归纳和选题生成。
4. 短时凭证缺失不阻塞正文分析；互动数据稍后补齐时，更新数据集版本和来源时间。

## 访问状态不是模式

不再以“登录 / 不登录”划分用户功能，因为这会掩盖两种凭证的用途和时限。

| 访问状态 | 来源 | 有效范围 | 用途 |
| --- | --- | --- | --- |
| 公开访问 | 文章 URL | 无会话依赖 | 下载公开正文和图片 |
| 后台授权 | Exporter QR / auth-key | 约 4 天 | 发现公众号、同步历史列表、订阅管理 |
| 微信会话 | 用户打开文章后的短时凭证 | 约 25 分钟，同一 `__biz` | 互动指标、精选评论、备用历史采集 |

正文归档是共享的交付能力。单篇文章按以下顺序选择证据和内容来源：

1. 已收藏的微信页面快照。
2. 已有本地 Markdown 和图片。
3. 公开 URL 下载。
4. 经用户明确确认的微信会话请求。
5. 失败后保留文章元数据和失败原因，允许稍后重试。

不得把后台授权或微信会话凭证用于未声明的用途，也不得在日志、Markdown 或导出目录保存原始凭证。

## 统一架构

```text
Exporter 控制面 ─────> 文章范围 / 标题列表 / 订阅状态
                           │
             ┌─────────────┴─────────────┐
             ▼                           ▼
公开 URL 内容面：正文下载          微信收藏数据面：凭证 / 快照 / 互动
             └─────────────┬─────────────┘
                           ▼
                    本地资料库 / 数据集清单
                           ▼
                    外部 Wiki / 分析机制
```

### 共享资料库层

资料库是所有来源的唯一落点，而不是按模式建立平行目录。

- 账号以稳定的 `__biz` / `fakeid` 和规范化名称关联。
- 文章以规范化 URL、`appmsgid`、`idx` 和本地文章键去重。
- 可见交付仍在 `~/Downloads/wechat-articles/<公众号名>/`，包含 `articles/`、`images/`、`index.csv` 和文章关联快照目录。
- `exporter.sqlite` 继续承载订阅和文章元数据；不要再创建第二个“微信收藏数据库”。
- wizard 的任务、锁和运行记录仍留在其现有 SQLite 中；它们是编排状态，不是内容事实源。

### 微信收藏会话层

将现有增强代理演化为常驻 `collection service`。首次启用时从 `23032-24045` 选择端口并保存；正常重启复用该端口和固定 upstream，不在每次任务开始时重新切换系统代理。服务由三个可独立工作的能力组成：

1. **article-save**：注入“收藏到本地”，接收当前页面 DOM 快照。
2. **credential-capture**：识别文章所属公众号，捕获短时互动请求所需的凭证。
3. **history-capture**：仅在备用历史列表工作流中观察 `profile_ext/getmsg`。

同一设备只能有一个活动收藏端口。所有重启必须沿用当前端口与 upstream，先恢复安全路由、再重启、最后重新切回，避免死端口。

常驻是目标默认值，但不能直接把当前会话进程改成无限运行。上线前必须具备：

- launchd `KeepAlive` 或等价守护，确保本地代理先存活、系统路由后启用。
- 心跳和 watchdog；进程不可达时立即重启，连续失败时恢复此前系统代理。
- 非微信流量只透传到启动时固定的 upstream，不记录请求正文、查询参数或凭证。
- 原始短时 credential 仅保存在守护进程内存中，按 `__biz` 隔离并在到期时清除；SQLite 只记录状态和时效。
- 提供“暂停微信收藏”和“禁用微信收藏”，暂停只恢复路由，禁用同时移除常驻服务。

在这些保护完成前，当前手动 start/finish 会话继续保留，不能仅为减少一次操作就让系统长期指向无守护的进程。

### 互动数据同步层

批量互动同步不是对所有订阅的自动后台任务，但可以由一个明确的上层任务自动触发。例如用户已经说“下载这个公众号最新 20 篇并补充互动数据”，不需要在拿到凭证后再次确认。

1. 根据当前微信文章识别 `__biz`，关联本地目标账号。
2. 检查账号是否已有同步文章；没有时提示先用订阅同步或备用历史采集建立文章列表。
3. 展示计划范围，例如最近 20 篇或某日期区间，以及凭证剩余时间。
4. 凭证缺失时进入 `waiting_credential`，复制一篇目标文章 URL 并等待用户在微信打开；正文下载继续执行。
5. 逐篇确保有文章标识、`comment_id`、`appmsgid` 和 `idx`；缺失时只记录不可获取原因，不猜测。
6. 使用本地受控请求链抓取文章指标、精选评论和可得到的回复；最大并发为 2。
7. 以结构化数据写入资料库，并替换文章 Markdown 中的页面数据区块。

默认不使用公网 Exporter、公共代理池或第三方转发服务。若当前用户已配置上游代理，只允许沿用用户明确配置的上游；诊断必须显示“直连”或“用户配置的上游”，不得泄露凭证。

### 旧代理历史的作用

旧代理历史模式通过用户打开旧版 `profile_ext` 页面并滚动，观察实际发生的 `getmsg` 响应。它不再是一个产品模式，保留价值只有三项：

1. Exporter 后台授权不可用时，作为人工、被动的一次性历史列表兜底。
2. 微信接口或参数变化时，作为真实网络证据源，帮助修复主动同步实现。
3. 兼容旧命令、已有运行状态和故障恢复流程，避免升级后破坏用户环境。

主动历史同步是后续独立能力，未实现和验证前不得当作当前 fallback。旧 `history-capture-*` 保留两个发布周期并退出 README/SKILL 主流程，之后仅作为诊断命令。

## 数据模型调整

现有 `articles`、`article_comments`、快照和 Markdown 页面数据区块继续复用。新增或调整的数据应有明确来源和完整性状态。

| 实体 | 最小字段 | 说明 |
| --- | --- | --- |
| `wechat_collection_service` | service_id, port, upstream_kind, enabled_at, state | 常驻服务生命周期和恢复状态；不存 credential |
| `article_credentials` | account_id, captured_at, expires_at, status | SQLite 只保留状态元数据；原始 credential 仅存在于守护进程内存中 |
| `engagement_runs` | run_id, account_id, scope, requested_count, success_count, failed_count, credential_expires_at | 可审计的批量同步运行记录 |
| `article_metrics` | article_id, captured_at, source, read_count, like_count, old_like_count, share_count, comment_count | 指标时间序列；不虚构收藏数 |
| `article_comments` | article_id, comment_id/content_id, comment_scope, source, fetched_at | `comment_scope=elected` 必填，避免被误解为全量评论 |
| `article_comment_replies` | comment_id, reply_id, source, fetched_at, complete | 评论回复独立存储，记录是否可能截断 |
| `dataset_manifests` | dataset_id, account_id, article_scope, version, created_at | 供外部 Wiki/分析机制稳定读取本地数据，不承担分析本身 |

页面快照和 API 同步的数据必须共存，不互相覆盖原始证据。Markdown 仅呈现当前最佳值，并在每个字段标出来源：`page_snapshot`、`wechat_session_api` 或 `missing`。

## 用户交互

### 订阅同步

```text
用户：订阅并同步「某公众号」
系统：检查后台授权 -> 搜索/选择账号 -> 同步文章列表 -> 展示可选文章
用户：下载最近 20 篇
系统：将选中正文归档到本地资料库
```

### 微信收藏和互动补充

```text
用户：下载「某公众号」最新 20 篇并补充互动数据
系统：Exporter 同步并展示 20 篇标题；正文下载立即开始
系统：当前公众号短时凭证缺失，已复制其中一篇文章链接
用户：在微信中打开这篇文章
系统：自动识别公众号和短时凭证；恢复 20 篇互动任务
用户：需要保存当前页面时，可额外点击“收藏到本地”
系统：保存当前文章的高保真页面快照
系统：写入正文、可用指标和精选评论，并说明评论范围
```

### 备用历史列表

```text
用户：从微信获取这个公众号的历史文章
系统：说明这是备用路径 -> 启用 history-capture -> 用户在微信打开旧版历史入口并滚动
系统：保存历史列表 -> 让用户选择 -> 归档选中文章
```

## CLI 和兼容策略

新增用户语义命令，现有技术命令保留一个版本周期作为别名：

| 新命令 | 兼容旧命令 | 行为 |
| --- | --- | --- |
| `wechat-collection-enable` | 新增 | 首次启用常驻服务和路由 |
| `wechat-collection-status` | `proxy-enhancer-status` | 查看会话、入口和凭证元数据状态 |
| `wechat-collection-pause` | `proxy-enhancer-session-finish` | 暂停路由但保留配置 |
| `wechat-collection-resume` | `proxy-enhancer-session-start` | 恢复常驻收藏路由 |
| `wechat-collection-disable` | 新增 | 恢复系统代理并移除常驻服务 |
| `wechat-collection-sync-engagement` | 新增 | 为指定账号/范围同步指标和精选评论 |
| `wechat-collection-history` | `history-capture-*` 编排 | 备用获取历史列表 |
| `library-import` | `download-url` / wizard URL mode | 已知链接归档 |

Skill 和 README 只展示新命令及自然语言入口。旧命令的 help 文案必须标记为“兼容命令，请使用微信收藏”。所有命令保留 JSON 输出，新增字段必须向后兼容。

## 实施阶段

### 阶段 1：产品命名和路由

- 将 README、SKILL、wizard 提示的“代理增强”改为“微信收藏”。
- 重新组织首页：订阅同步、微信收藏、链接导入。
- 旧代理历史移动到微信收藏的备用能力说明。
- 新增 CLI 别名，不删除已有自动化入口。

### 阶段 2：正文与凭证并行编排

- Exporter 返回目标范围后立即启动公开正文下载。
- 增加账号级 credential 状态机和任务 `waiting_credential` 状态。
- 缺失时自动复制代表文章 URL；捕获后只恢复对应 `__biz` 的等待任务。
- 生成供外部分析使用的数据集清单，不在本 Skill 内生成选题。

### 阶段 3：安全常驻微信收藏

- 抽取会话状态，明确 `article-save`、`credential-capture`、`history-capture` 三项能力。
- 复用当前随机端口、WebView 重置、`Cache-Control: no-store`、诊断和安全重启机制。
- 增加 launchd 守护、心跳、异常恢复和暂停/禁用命令。
- 原始短时 credential 只在内存中存活，不落盘。
- 验证崩溃、开机、网络服务切换和重启时系统代理始终可恢复。

### 阶段 4：本地批量互动同步

- 为当前公众号建立临时 credential broker。
- 实现文章指标读取、精选评论分页、受限回复读取和最大并发 2 的队列。
- 将运行记录、字段来源、评论范围和完整性写入 SQLite。
- 把结果合并进已有 Markdown 页面数据区块和指标历史。

### 阶段 5：统一资料库和下载策略

- 让 Exporter、微信收藏、链接导入统一按账号/文章键关联。
- 明确正文来源优先级和重试策略。
- 让收藏保存的文章无需再次“下载”即可进入资料库。
- 保留 URL 下载作为公开链接和失败兜底。

### 阶段 6：旧代理历史收敛

- 优先实现短时凭证下的主动历史同步。
- 在两个发布周期后，将 `history-capture-*` 和其他旧技术命令移入兼容/诊断层。
- 保留 `proxy-snapshot-*` 仅用于诊断，不写入主流程文档。
- 仅在 Exporter 和主动历史同步都失败时，才展示旧版历史页面滚动流程。

## 验收标准

- 用户不需要理解“代理增强”即可完成一次微信收藏。
- 常驻守护、崩溃回滚和网络服务切换通过真实 macOS 验收后，微信收藏才默认常驻；此前普通任务继续使用显式会话。
- 用户打开一个公众号文章后，能看到该公众号短时数据权限是否可用、何时过期、可补充的文章范围。
- 公开正文下载不被短时凭证阻塞，捕获凭证后等待中的互动任务可自动恢复。
- 在明确选择范围后，系统能为同一公众号的已同步文章写入可验证的互动指标和精选评论。
- 所有评论输出都标记为 `elected`，所有回复都标明是否完整；不得承诺全量评论。
- 没有任何默认公共代理或公网 credential 转发。
- 保存当前文章、批量互动同步、订阅同步和链接导入都落到同一账号目录和文章记录。
- 主动暂停、异常退出和安全重启后，系统代理可恢复；诊断不展示废弃端口 `8899`。
- Wiki、外部同步、内容改写和 SaaS 不进入本期范围。

## 测试策略

- **迁移测试**：新命令和旧别名输出同一会话状态，随机端口范围仍为 `23032-24045`。
- **常驻测试**：覆盖 launchd 重启、进程崩溃、端口占用、上游变化、系统网络服务切换和 watchdog 恢复。
- **安全测试**：SQLite、日志、快照和 Markdown 中不出现 raw cookie、`key`、`uin`、`pass_ticket` 或 `appmsg_token`。
- **范围测试**：凭证只能用于匹配的 `__biz`；过期后拒绝批量同步并提示重新打开文章。
- **编排测试**：正文下载与 `waiting_credential` 并行，凭证到达后只恢复匹配公众号的互动任务。
- **数据测试**：顶级评论分页、精选评论范围、回复截断、字段来源和 Markdown 替换均有 fixture 覆盖。
- **资料库测试**：同一篇文章来自链接导入、订阅同步、微信收藏时不产生重复 Markdown 或重复页面数据块。
- **端到端手测**：启用常驻收藏、打开文章捕获凭证、保存页面、补充最近 N 篇、暂停路由和恢复系统代理。

## 非目标

- 不抓取或承诺全部用户评论。
- 不绕过登录、付费、私密文章、平台权限或风控。
- 不在没有明确任务范围时自动对所有订阅账号批量抓互动数据。
- 不使用第三方公共代理转发微信凭证。
- 不实现外部 Wiki 写入、云端同步、内容改写或 SaaS 服务。
